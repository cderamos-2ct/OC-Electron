# Incoming CTO Learning Operating Package

## Purpose
Turn the learning-system direction Christian approved into an operating package that can actually be used day to day.

This package assumes:
- CTO-first focus
- adaptive expansion into adjacent fields as projects appear
- all useful content formats matter
- everyday learning is the target, but flexibility is required
- outputs must exist in both dashboard form and durable reference docs

---

## 1) Daily cadence model

### Default daily mode
Use this when the day is reasonably normal.

#### Pomodoro 1 — Intake (25 min)
Pick one focused source packet:
- internal source docs
- AI/software research
- relevant podcast/video/article cluster

Goal:
- consume one meaningful chunk
- not ten random tabs

#### Pomodoro 2 — Synthesis (25 min)
Convert intake into one or more of:
- short brief
- NotebookLM pack update
- questions / implications
- experiment idea
- team-shareable note

Goal:
- nothing stays as passive consumption only

### Busy-day mode
Use this when operations are chaotic.

Minimum viable learning:
- one audio/podcast-style item during drive/work
- one short capture afterward:
  - what mattered
  - what changed
  - what to revisit

### Deep-work mode
Use this when the topic is strategically important.

Add:
#### Pomodoro 3 — Execution (25 min)
Turn the synthesis into:
- experiment brief
- partner memo
- architecture note
- backlog item

---

## 2) Weekly rhythm

### Weekly outputs
At least once per week, produce:
- **CTO Weekly Brief**
- **Future-Forward Experiments Update**
- **What Changed This Week** summary

### Weekly review questions
- what did Christian learn that matters strategically?
- what changed in the market/AI/software landscape?
- what should the dev team test next?
- what needs partner visibility?
- what is noise and should be dropped?

## 4) NotebookLM pack structure

### Notebook 1 — CTO Core
Purpose:
- top-level thinking about AI, software, platform, service model, and CTO role

### Notebook 2 — Clarity / IDP / ServFlow Core
Purpose:
- internal source docs
- SOWs
- diagrams
- architecture understanding

### Notebook 3 — Future-Forward AI / Software Research
Purpose:
- curated external research on agents, orchestration, evals, multimodal systems, voice, automation, workflow intelligence, etc.

### Notebook 4 — Experiment / Partner Brief Pack
Purpose:
- material that turns into partner briefs, proposals, and dev-team experiments

### Pack rule
Do not create giant junk-drawer notebooks.
Each notebook should have a clear purpose and source boundary.

---

## 5) Source priority model

### Priority 1 — Internal source truth
Always start here when trying to understand what matters.
- SOWs
- diagrams
- proposals
- architecture notes
- real operating constraints

### Priority 2 — High-signal external research
Only curated, highly relevant material.
- best podcasts
- best YouTube/talks
- best essays/newsletters
- best papers/repos when actually useful

### Priority 3 — Market / competitor / product signals
Use to stay future-forward, not to get hypnotized by hype.

## 6) Dashboard fields

The dashboard should eventually expose at least:

### Learning status
- current focus domain
- current notebook / source pack
- last learning session
- next recommended session

### Research queue
- what to consume next
- why it matters
- format (audio / video / article / doc)

### Output queue
- brief to write
- experiment to define
- memo to share
- team artifact to produce

### Understanding progress
- internal sources reviewed
- open questions
- current working thesis

### Experiment pipeline
- ideas captured
- ideas refined
- ideas assigned
- ideas tested

---

## 7) Adaptive expansion rule

When a new project/workstream appears, create a temporary learning branch:
- domain context
- key sources
- relevant technologies
- likely experiment ideas
- partner/team outputs needed

Then either:
- merge it into the core system if it becomes durable
- archive it if it was one-off

This prevents the CTO system from bloating permanently every time a random project appears.

## 8) Output standards

### Audio output
Best for:
- driving

- review/reinforcement

### Short brief
Best for:

- partner alignment
- personal review

### Reference doc
Best for:
- durable lookup
- architecture/business understanding
- later synthesis

### Experiment brief
Best for:
- dev team action
- testing future-forward ideas quickly
